# TaskTree 系统架构说明文档

本文档旨在全面梳理 TaskTree 项目的整体架构、技术栈、核心模块设计以及数据流向，帮助开发者快速理解系统内部运作机制。

---

## 1. 系统总体架构

TaskTree 采用了经典的**前后端分离架构**，致力于为用户提供一个跨平台、高性能、直观可视化的项目与任务管理系统。

### 1.1 技术栈总览

| 分层 | 核心技术 | 说明 |
| :--- | :--- | :--- |
| **前端 (Client)** | React 18, TypeScript, Vit

e | 构建用户界面的核心框架，使用 Vite 提供极速的本地开发热更新。 |
| **UI 与交互** | Ant Design 5, Tailwind CSS | AntD 提供标准化的组件基础，Tailwind 用于快速编写原子化辅助样式。 |
| **复杂视图生态** | `@dnd-kit`, `reactflow`, `gantt-task-react` | 支撑无限层级拖拽、依赖梳理网状图和时间线调度的核心图形库。 |
| **状态管理** | Zustand | 轻量级状态管理，主要负责全局的用户认证态及部分需要跨同级组件共享的数据。 |
| **后端 (Server)** | Python 3.11, FastAPI | 高性能、基于标准的现代 Python Web 框架，自带交互式 API 文档。 |
| **数据持久化** | SQLAlchemy 2.0 (ORM), SQLite(异步) | 采用 2.0 风格的声明式 ORM 进行对象映射，通过 `aiosqlite` 提供全异步的数据库访问。 |
| **安全认证** | JWT, passlib(bcrypt) | 无状态的 JSON Web Token 标准校验访问权限，密码使用 bcrypt 单向哈希。 |

### 1.2 高层数据流向

1. **用户侧**：通过 Web 浏览器或 Tauri 桌面端壳子访问前端 Vite 打包的静态资产。
2. **前后端通信**：前端封装 Axios 实例 (`src/api/index.ts`)，统一携带 Bearer Token 并对异常做全局拦截提示。通过 RESTful API 与后端的 `/api/v1/*` 路由通讯。
3. **后端业务流**：FastAPI 路由层接收校验参数 -> 注入数据库的单例 AsyncSession -> 核心数据查询及权限验证 -> 使用 Pydantic Schemas 进行数据过滤和序列化 -> 响应给前端。

## 2. 后端架构设计 (Backend)

后端代码位于 `backend/app` 目录下，遵循典型的 MVC/三层架构模式演变而来的领域模块化结构：

### 2.1 目录结构

```text
backend/app/
├── api/
│   └── v1/            # 路由层：按业务拆分为 auth.py, tasks.py, projects.py 等
├── core/              # 核心基座层：配置 config.py, 数据库配置 database.py, 安全鉴权 security.py
├── models/            # 数据库模型层：SQLAlchemy 表定义 mapped classes
├── schemas/           # 数据传输模型层：Pydantic models，负责校验入参和格式化出参
└── main.py            # 应用程序入口，组合所有路由并挂载 CORS 等中间件
```

### 2.2 核心机制说明

- **数据库引擎延迟加载**：在 `database.py` 中，使用了全局单例模式的 `create_async_engine` 和 `async_sessionmaker`。在 FastAPI 的 `Depends(get_db)` 依赖注入中产出 Session，并自动利用 `try...except...rollback` 保障了事务和连接释出的安全性。
- **状态流转与权限控制**：
  - 接口粒度的防御：每个需要操作实体的路由，都必须调用一次类似 `get_task_with_access()` 的函数。这保证了即使用户枚举到了 ID，也无法访问不属于自己名下的记录。
  - 所有的常量（包括角色权限和有效状态流转转移矩阵）统一维护在 `core/constants.py` 中。
- **关联表设计 (Models)**：使用 `DeclarativeBase` 定义了 10 张核心表，高度利用了 SQLAlchemy 的 `relationship` 的延迟加载外键链。尤其是 **Task 模型使用自引用** (`parent_id`) 构建了一对多的任务树结构。

## 3. 前端架构设计 (Frontend)

前端代码位于 `frontend/src`，架构重点在于极复杂界面的切割与状态同步。

### 3.1 目录结构

```text
frontend/src/
├── api/               # 对 Axios 的二次封装及所有后端端点的调用的中心化文件
├── components/        # 公用的独立可复用 UI 与视图组装区块
│   ├── view/          # 高阶可视化视图（Gantt）
│   ├── dependency/    # 依赖连线拓扑图（ReactFlow）
│   ├── task/          # 任务详情浮层等业务组件
│   └── export/        # 功能类弹窗
├── constants/         # 前量相关的静态配置（与后端对齐）
├── pages/             # 路由顶层页面：路由的实际挂载点
├── stores/            # Zustand：存放身份认证 authToken 和用户信息
├── types/             # 全局的 TS Interface 声明
└── App.tsx            # 基于 react-router-dom 的页面骨架与路由导航
```

### 3.2 界面视图解耦策略

任务树项目最大的前端挑战在于数据的**多种形态表现**。我们使用了以下机制解耦：
1. **单一数据源**：`ProjectDetail.tsx` 是数据的总管家，负责通过 `tasksAPI.getTree()` 拉取结构化的全量数据。
2. **扁平化能力映射**：针对某些仅接受列表不接受树的呈现库（如 Kanban 或 List 或 Graph），在传入之前，通过纯函数 (`flattenAllTasks`, `collectDependencies`) 在当前闭包计算生成展平的形态对象，投喂给对应的子组件。
3. **独立渲染器模式**：
   - *树形视图 (Tree/dnd-kit)*：直接渲染组件递归树。
   - *甘特图 (gantt-task-react)*：将开始/结束时刻映射为时间条。
   - *依赖图 (ReactFlow)*：将 Task 映射为 Node，`task_dependencies` 映射为 Edge，由专有的布局算法引擎绘制连线。

## 4. 关键业务流程解析

### 4.1 无限层级任务树加载与渲染
1. 客户端访问某项目。
2. 后端接收 `GET /projects/{id}/tasks/tree`：
   - 将这百余个平铺行的表记录一次性查出。
   - 使用 **HashMap** 在内存中极速拼装父子级。由于提前根据 Sort_Order 排过序，此步骤能保证 O(n) 的复杂度生成 JSON 嵌套树返回。
3. 客户端接收到深层 Object，利用 React 组件的自我递归 (`<SortableTaskItem>`) 一层层套娃渲染展现。

### 4.2 任务状态与通知联动
1. 用户在前端切换一个子任务的状态为「已完成」。
2. 调用 `PUT /tasks/{id}`。
3. 后端路由除了执行正常的写入，还会判定：
   - 是否符合 `constants.py` 规定的流转路径。
   - 如果发生实质改变，调用 `create_notification` 给此任务负责人注入一条站内通知信息至 `notifications` 表，并随同本次更新事务一并 Commit 到硬盘。
4. 客户端再次重绘任务卡片，展示划线样式。

## 5. 项目未来迭代留口

为了保持架构健壮以应对后续可能的大型演进，系统目前具有以下设计冗余：
- **Schema 的分离**：在接口的数据验证层面使用了严格的进出剥离（如分为 `TaskCreate`, `TaskUpdate`, `TaskDetailResponse`），使得表结构可以激进修改而不影响外抛接口契约。
- **统一化鉴权入口**：`get_current_user` 依赖，未来如需抛弃 JWT 换 Session，或接入第三方 OAuth 中心，可实现零成本侵入。
- **独立附件与日志模块**：虽然 `upload` 尚未打通业务，但底层实体表 `TaskAttachment` 与 `OperationLog` 均已准备就绪，待前端接口挂载后即可激活。
